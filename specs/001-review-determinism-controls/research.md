# Research: Review Determinism Controls

## Decision: Review mode is a three-value workspace setting

**Decision**: Add a persistent workspace review mode with values:

- `skip`: do not start review checks and do not block.
- `log`: run review checks and emit review data/log output, but do not block.
- `normal`: run review checks and block according to accepted findings.

The absent setting is equivalent to `normal`.

**Rationale**: This matches the clarified product behavior while preserving
current behavior for existing users. It separates "do not spend review work" from
"collect review data without enforcement", which are materially different
rollout modes.

**Alternatives considered**:

- Boolean enabled/disabled: rejected because it cannot express logging-only mode.
- Command-only temporary bypass: rejected because the requested behavior includes
  permanent skip.
- Team-admin-only switch: rejected because the product must support individual
  restricted workstations.

## Decision: Store mode in project-local generated configuration

**Decision**: Treat review mode as workspace-local configuration under
`.gitpushreview/config/`, generated by `init` and read by `check`, `doctor`, and
diagnostic commands.

**Rationale**: GitPushReview already creates project-local config and rules under
`.gitpushreview`. This keeps user-mode operation independent of global
installation and makes the effective mode inspectable in the repository.

**Alternatives considered**:

- Environment variable only: rejected because permanent skip/logging mode needs
  discoverable persistent state.
- Overload model configuration: rejected because review mode is not model
  configuration and should remain meaningful even in `skip` mode.
- User home directory config: rejected for this feature because it complicates
  multi-repo precedence and is not required by the clarified user-mode scope.

## Decision: Candidate set remains the authoritative finding boundary

**Decision**: The deterministic review candidate set produced by rule routing is
the only set of rule identifiers that can produce accepted findings. Model
findings outside that set become rejected diagnostics and do not affect accepted
finding IDs, scores, or block status.

**Rationale**: The reported instability is caused by inconsistent triggered
rules for identical staged input. A deterministic candidate boundary creates a
local invariant that can be tested without relying on model quality.

**Alternatives considered**:

- Trust high-score out-of-candidate model findings: rejected because it recreates
  nondeterminism and lets the model override routing.
- Fail the review when the model emits out-of-candidate findings: rejected
  because it would punish users for model behavior and create noisy retries.
- Silently discard rejected findings: rejected because diagnostics are needed to
  debug weak model behavior and route gaps.

## Decision: Stabilize ordering and accepted finding normalization

**Decision**: Candidate rules, accepted findings, rejected findings, and
diagnostic output should be ordered deterministically using route/rule order and
stable secondary keys. Decision calculation must use accepted findings only.

**Rationale**: Stable sets alone are not enough for repeatability tests and
human-readable diagnostics. Deterministic ordering reduces apparent churn and
makes repeated runs easier to compare.

**Alternatives considered**:

- Preserve model output ordering: rejected because model ordering can vary.
- Sort only final reports: rejected because JSON diagnostics and tests also need
  deterministic order.

## Decision: User-mode install means repository-local operation

**Decision**: Support initialization and operation when users can write the
repository but cannot write global package locations or protected system paths.
If automatic hook installation is unavailable, provide Chinese guidance for a
manual check path or repository-local hook path.

**Rationale**: This maps to the clarified scope and common enterprise desktop
constraints without introducing cross-repository user profiles.

**Alternatives considered**:

- Require global install: rejected by the feature.
- Implement user-home shared configuration now: rejected as broader than needed.
- Auto-switch among global/user/repo installation modes: rejected as higher
  operational complexity for this feature.

## Decision: Expand Chinese help as a CLI contract

**Decision**: General help, command-specific help, invalid usage, and
unknown-command messages must explain command purpose, options, and examples in
Chinese.

**Rationale**: Help output is the recovery surface for installation/configuration
problems. Treating it as a contract enables focused tests without changing the
CLI architecture.

**Alternatives considered**:

- README-only documentation: rejected because users need inline help at the
  moment of failure.
- General help only: rejected because option-level behavior is part of the
  clarified requirement.

## Decision: Bad-smell target is `openmole@0.8.2` from `agiledon/bdr` main

**Decision**: Use `openmole@0.8.2` from `https://github.com/agiledon/bdr.git`
main as the maintainer-approved fixed target for this plan. The repository head
observed during planning was `0acd889777f175c46efec2be20ae3ed538ccbb3e`.

**Rationale**: `npm view agile-bdr` returned registry 404 with an unpublished
notice, while the GitHub repository currently exposes package metadata for
`openmole@0.8.2`. A fixed git-source target is reproducible and matches the
clarification that the target version must be approved and recorded during
planning.

**Alternatives considered**:

- `agile-bdr@latest` from npm: rejected because the package is unavailable in
  the registry.
- Keep vendored `agile-bdr@0.5.0`: rejected because the feature asks to update
  the bad-smell dependency.
- Floating GitHub main: rejected because it is not a fixed target.

## Decision: Validation remains local and stubbed

**Decision**: Use existing local test seams for validation: `routeCommand()` for
CLI, `modelInvoker` for review runner behavior, `fetchImpl` for model client,
and temp Git repositories for staged-data behavior.

**Rationale**: This avoids real model/network dependence and keeps regression
tests deterministic.

**Alternatives considered**:

- Run a live model to prove review stability: rejected because live model output
  is not a deterministic test fixture.
- Snapshot entire human-readable output only: rejected because JSON diagnostics
  and accepted/rejected finding sets need direct assertions.
